High level Requirements

Let's Discuss Slack like Chat Application

  • User's can chat 1:1
  • User can create Room/Group
  • Room can have multiple users
  • Users can search user and rooms

Follow up Questions

  • To understand expected load on the system:
    • What is number of messages exchaged every minute
    • What is size of the messages
    • Calculate the bandwidth requirements for the system i.e. number of messages exchanged per second
    • These questions and requirements also gives idea of the scale of the system
  • To understand how much and type of storage we may need:
    • Discuss the data model of the system to understand the type of data storage needed.
    • Discuss what type of db, MySQL or NoSQL is needed and why.
    • For how many years we need to store the messages to understand scaling of db.

Components of System

  • UI Client - Web or Mobile App
  • Server - having backend service
  • Load Balancer
  • Database - storage for users and messages
  • Cache (if needed in the application, for e.g. Chat application is read heavy so its good idea to have cache)

Details of Components

UI Client:

  • Based on the requirement we can have Web or Mobile(for e.g. Android, IOS) as client for our backend service
  • This component communicates with service using some protocol. For e.g we can have communication through REST API over HTTP

Data Model and Database:

  • Once the data model is discussed, you know the data storage needed.
  • For e.g in Chat System, we can use MySQL to store User info, Room info and relation between room and user and store messages with room id/ user id and sender.(This is just an example not necessary right way)
  • To store images etc in messages we can use S3 like datastore

Services/API and Server:

  • Discuss the API's used in the system
  • For example in Chat Application you will need API to find group, find user, create room, sign up and use WebSockets or Long Polling for sending and retrieving messages.
  • This application is deployed in Server and based on follow up question we might need to scale the server to handle the load
  • Horizontal scaling of server is one approach i.e. adding more servers with our service, this introduces the need of Load Balancers

Load Balancer:

  • Load Balancer are used to spread traffic across a cluster of servers and track status of all the resources.
  • They can be used at each layer to utilize scalability and redundancy.
  • Some of the common Load balancing Algorithms you want to read about are:
    • Round Robin
    • Weighted Round Robin
    • Least Connection
    • Weighted Least Connection
    • Least Bandwidth
    • Least Response Time
    • IP Hash

Cache:

  • Cache helps to reduce the latency involved in requests to database.
  • Frequently requested data can stored in cache
  • Good to know some of the Cache Eviction Policies:
    • FIFO - First In First Out
    • LIFO - Last In First Out
    • LRU - Least Recently Used
    • MRU - Most Recently Used
    • LFU - Least Frequently Used
    • RR - Random Replacement
  • Good to know about some of Cache Invalidation Techniques:
    • Write-through Cache
    • Write-around Cache
    • Write-back Cache
